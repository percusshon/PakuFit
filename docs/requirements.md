# PakuFit 要件定義（MVP）

## 想定ユーザー

- 忙しくて日々の食事を継続記録しづらい成人ユーザー
- コンビニ/外食中心でも、1日の記録を崩れずに続けたいユーザー
- 極端なダイエットより、日常のバランス感覚を育てたいユーザー
- 体重・体調データは持つが、医療的助言は求めないユーザー

## ユースケース

1. 写真を撮って食事を記録し、AI推定を起点に入力を簡略化する
2. 推定内容（料理名・量・食べた割合・調理方法）をユーザーが補正する
3. 日内の摂取カロリーとPFCの合計を確認し、残量感を把握する
4. 目標に合わせて次の食事候補を選ぶ
5. 将来の在庫・価格・提携情報連携を想定しつつ、まずは中立な候補提示を行う

## MVP要件

### 1) ユーザー基盤

- メールまたはSNS等での本人登録・ログイン（認証方式は後段で実装）
- ユーザーごとの目標保存
- 利用規約同意および未成年向け注意喚起

### 2) 食事インプット

- 食事写真のアップロード
- 食事テキスト入力
- 同一食事に対して写真とテキストの両方が可能

### 3) AI推定と補正

- AIが料理名・食材候補を推定
- 推定は**不確実**であることをUI上で明示
- ユーザーは必ず料理名・量・食べた割合・調理方法を補正可能
- 補正履歴を保存し、再推定精度の改善に使える前提にする

### 4) 栄養情報

- カロリー/PFCを計算して**概算表示**
- 表示は合計、食事別、直前までの累積
- 「推定」「目安」「ユーザー確認が必要」を明示

### 5) 1日サマリー

- 日別の合計カロリー、PFC、残り目安
- 食事数、食事候補履歴
- 目標との差分表示

### 6) 食事候補提案

- 時間帯、履歴、残量、希望（外食/自炊/コンビニ）、予算を入力して候補提示
- 候補はあくまで**次の食事候補**として提示
- 健康効果の断定は禁止、必要時に注意喚起を表示

### 7) 安全・プライバシー

- 食事写真・体重・履歴は個人に紐づくセンシティブ寄りデータとして扱う
- RLS（行レベル）を前提に、本人のみ参照できる構成
- 利用中の健康不安には専門家相談を促す

### 8) 法務・ブランド

- 商標一次確認の実施予定を文書化
- 弁理士レビュー前提を明記
- サービス内での断定的健康表現を排除

## 将来要件

- JAN/バーコード読取連携
- コンビニ/スーパー商品DB、地域・在庫・価格を加味した候補提示
- 提携導線（広告・アフィリエイト）を透明性付きで導入
- 将来のアレルゲン・栄養アレルート対応拡張
- 専門家監修コンテンツ連携（診断ではない説明記事）

## 成功条件

- 1日入力が負荷低く5~10分以内に完了できる
- AI推定後にユーザー補正率（主要項目の確認/修正）を把握できる
- 候補提示が「選択」ではなく「判断材料」として認識されるUIを実現できる
- 医療・疾病改善の断定表現が仕様内に混入しない
- 未確認点を明示しつつ、公開前の法務・セキュリティレビューで主要懸念が潰せる

## やらないこと

- 医師・栄養士に代わる診断、治療、疾病改善の個別助言
- 「この食事で痩せる」等の結果保証
- 自動で過度なカロリー制限を提案する挙動
- 実際の広告最適化・収益化ロジック（初期MVPではオフ）
- 実API接続された写真Vision推定（初期は設計とUI/フロー中心）
