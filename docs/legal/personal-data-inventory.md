# 個人データ取扱台帳（現状整理）

> 目的: 弁護士が個人情報保護法の観点でレビューできるよう、PakuFit が実際に取得・保管・送信するデータを実装ベースで一覧化したもの。
> 出典は `supabase/migrations/`（DBスキーマ）と `lib/ai/`・`app/api/`（外部送信）。
> **本書は事実整理であり、法的評価は弁護士レビューを前提とする。**

## 1. 取得する個人情報・関連データ

| データ区分 | 保管先（テーブル等） | 主な項目 | 個人情報該当性（暫定） | 備考 |
| --- | --- | --- | --- | --- |
| アカウント | Supabase Auth `auth.users` | メールアドレス（Magic Link 認証） | 該当 | パスワードは保持しない（メールリンク認証） |
| プロフィール | `public.profiles` | 表示名、プロフィール種別 | 該当しうる | 任意入力 |
| 健康目標 | `public.user_goals` | 目標カテゴリ、目標カロリー/PFC、メモ | 要配慮個人情報には当たらない見込みだが**健康関連で要確認** | 体重そのものの列は現状なし（メモに含む可能性あり） |
| 食事記録 | `public.meal_entries` | 食事ラベル、摂取日時、食事説明、概算栄養値、入力経路 | 該当（生活情報） | |
| 食事写真 | `public.meal_photos`（Storage `meal-photos`） | 画像のパス/URL、メモ | 該当（**写り込みで顔・所在等の可能性**） | 画像実体は Supabase Storage |
| 推定食品 | `public.food_items` | 推定料理名・量・栄養値 | 該当（生活情報） | |
| 栄養推定 | `public.nutrition_estimates` | 概算値、信頼度、確定フラグ、`source_metadata`(jsonb) | 該当 | |
| 日次サマリー | `public.daily_nutrition_summaries` | 日別の概算栄養合計 | 該当 | |
| 提案履歴 | `public.meal_recommendations` | 候補名、理由、追従フラグ等 | 該当 | ルールベース。AIプロファイリングではない |
| 写真AI精度ログ | `public.photo_estimate_logs` | AI元概算・最終保存値・provider・信頼度 | 該当 | 追記専用・本人のみ(RLS)。**画像実体は含まない** |
| 監査ログ | `public.audit_logs` | 操作者ID、操作、対象、詳細(jsonb) | 該当 | 運用ログ |
| 商品マスタ | `public.partner_stores` / `public.partner_products` | 店舗・商品情報 | 非個人情報 | 事業者データ。本人データではない |

> 体重の継続的記録は設計方針（`privacy-and-safety.md`）に言及があるが、現行スキーマには独立カラムが無い。今後追加する場合は要配慮性の再評価が必要。

## 2. アクセス制御（安全管理措置の根拠）

- 全テーブルで **RLS（行レベルセキュリティ）** により本人の行のみ参照・更新可能。
- クライアントは **anon key のみ**を使用し、`service_role` キーは使用しない。
- 新規テーブル追加時は RLS ポリシーと `supabase/tests/` のアクセステスト追加を必須化（プロジェクト規約）。
- 認証は Supabase Magic Link。

## 3. 外部送信・第三者が関与するデータフロー

| フロー | 送信主体 | 送信内容 | 条件 | 法的論点 |
| --- | --- | --- | --- | --- |
| 認証・DB・画像保管 | サーバ/クライアント → Supabase | 上記1の全データ | 常時（サービス基盤） | Supabase を**委託先**として整理可能か。**保管リージョンが海外なら外国にある第三者への提供（28条）**の検討要 |
| 写真AI概算 | **サーバ**（`app/api/meals/estimate`）→ 外部AIサービス | 料理画像（base64）＋食事名ヒント（任意）＋固定プロンプト | **明示オプトイン同意時のみ**（`external_ai_consent`）。未同意はローカル擬似推定で外部送信なし | 外部AI事業者への提供が**委託 or 第三者提供**か。事業者が**米国等海外**なら28条。各事業者規約上の学習利用有無の確認要 |

- 写真AI送信は**サーバ側から**行い、ブラウザから直接外部へは送らない。
- アプリ側で**送信画像を保存・再利用しない**（画像の永続保存は写真AI経路では未実装。`meal_photos` への保存はユーザーが明示的に添付した場合の別経路）。
- 解析タグ・広告タグ・第三者Cookie等は現状**未導入**（→ 電気通信事業法 外部送信規律の該当範囲が現状限定的である点を確認したい）。

## 4. 保持期間・削除

- 退会・アカウント削除フロー: `auth.users` 削除時、ほぼ全テーブルが `on delete cascade`（一部 `set null`）で連動削除される設計。
- 明示的な保持期間ポリシーは**未策定**（事業者決定事項）。
- `photo_estimate_logs` は追記専用（更新ポリシー無し）。本人による削除は可能。

## 5. レビューで確認したい主な未確定事項

1. Supabase 本番リージョン（国内/海外）と、外国にある第三者提供（28条）への該当・対応方法。
2. 外部AI事業者の確定名・所在国・規約上の学習利用/保持の有無。委託 or 第三者提供 or 28条の整理。
3. `user_goals.notes` 等の自由記述に**要配慮個人情報**が混入した場合の扱い。
4. 食事写真への顔・所在等の**第三者情報写り込み**への対応（注意喚起で足りるか）。
5. 保持期間・削除運用ポリシーの要否と粒度。
6. 未成年（18歳未満）の利用可否と同意取得方法。
